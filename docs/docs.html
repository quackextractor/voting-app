3. Application Flowchart

Application traversal logic covering requirements F1 (Voting), F2 (Viewing), and F3 (Reset with validation).

flowchart TD Start((Start)) --> UserDecide{What does the user do?} %% Branch F1: Voting UserDecide -->|Selects option| Vote[Clicks 'Vote'] Vote --> ReqVote[POST /api/vote] ReqVote --> SaveDB[(Save to SQLite)] SaveDB --> RefreshData[Load new results] %% Branch F2: Viewing UserDecide -->|Just wants to see| View[Clicks 'View results'] View --> ReqGet[GET /api/results] ReqGet --> ReadDB[(Read from SQLite)] %% Branch F3: Reset UserDecide -->|Wants to reset| ResetInput[Enters Token] ResetInput --> ResetClick[Clicks 'Reset'] ResetClick --> ReqReset[POST /api/reset] ReqReset --> CheckToken{Is token OK?} CheckToken -- YES --> WipeDB[(Delete data in DB)] WipeDB --> RefreshData CheckToken -- NO --> Error[Return 403 error] Error --> ShowAlert[Show error message] %% Common output ReadDB --> ShowResults[Show Graphs] RefreshData --> ShowResults

6. API Endpoints

Method URL Endpoint Description and request/response body
GET /api/results Returns current voting states. No authorization is required.
Response: { "options": [{ "id": "a", "votes": 15 }, ...] }
POST /api/vote Saves a new user vote.
Body: { "optionId": "b" }
Response: { "success": true, "options": [...] }
POST /api/reset Resets all votes if the token matches the server configuration variable.
Body: { "token": "super_secret_key_123" }
Success response: { "success": true, "message": "Poll reset" }
Error response: { "success": false, "message": "Invalid token" } (Status 403)
